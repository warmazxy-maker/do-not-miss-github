# Agent 优化方向 Backlog

这份文档用于保存后续要逐项优化的 Agent 工程方向。当前只记录方向，不代表已经实现。

## 1. Query Understanding

把用户自然语言需求先解析成结构化标签和约束，再进入检索和推荐。

示例：

```json
{
  "intentTags": ["Java", "项目实践"],
  "timePreference": "weekend",
  "locationPreference": "大阪附近",
  "benefitPreference": "skill",
  "hardConstraints": [],
  "softPreferences": ["离我近", "适合初学者"]
}
```

目标：降低模型随机性，让检索、排序、解释都更稳定。

## 2. Context Compression

把历史活动、挑战、日志、聊天记录拆成长期画像、短期记忆和相关记忆，只把当前请求真正需要的上下文传给模型。

目标：控制 token、减少噪声、避免教练和推荐 Agent 被无关历史带偏。

## 3. Retrieval Rerank

先用混合召回拿到较宽的候选集，再用轻量模型或规则 reranker 判断候选与需求的相关性。

目标：解决“明明有相关事件但推荐为空”或“排序靠后很牵强”的问题。

## 4. Hallucination Guard

强化输出校验：

- eventId 必须存在于候选事件。
- 推荐理由不能出现数据库没有的组织、时间、地点。
- 不能把无报酬事件说成有报酬。
- 不能把非周末事件说成周末。
- 计划里的 EVENT 步骤必须来自候选事件。

目标：让 Agent 输出可控、可信、可解释。

## 5. Tool Use Trace

记录每次 Agent 调用的链路：

- 用户输入
- 工具调用结果
- 检索候选
- 模型输入摘要
- 模型输出
- 后端过滤结果
- 最终返回结果

目标：能排查推荐错误到底发生在 query 理解、召回、重排、模型输出还是后端校验。

## 6. Prompt Versioning

给每套 prompt 加版本号，例如：

- `event_recommendation_v1`
- `plan_agent_v1`
- `coach_log_v1`
- `self_analysis_v1`

目标：支持后续 prompt 优化对比和评测。

## 7. Evaluation Set

维护一批固定测试问题和期望结果，用来评估每次优化是否真的变好。

指标可以包括：

- TopK 命中率
- 幻觉率
- 低相关推荐比例
- 无效 eventId 数量
- 平均响应时间
- fallback 次数

## 8. Cost / Latency Optimization

减少不必要的大模型调用：

- 简单请求走规则推荐。
- 相同用户画像 + 相同需求做缓存。
- 只把 TopN 候选传给模型。
- 模型失败时 fallback 到规则结果。

目标：降低成本、减少等待时间、增强稳定性。

## 9. 运营审核后台

问题：社会端发布活动后如果直接进入学生端展示，会缺少内容质量控制、组织可信度控制和风险兜底。真实平台里，活动发布通常需要经过审核、下架、过期、拒绝等状态流转。

优化方向：

- 为活动增加 `PENDING_REVIEW`、`PUBLISHED`、`REJECTED`、`EXPIRED` 等状态。
- 社会端提交活动后默认进入待审核。
- 管理端可以审核通过、拒绝、下架活动，并填写原因。
- 学生端和 AI 推荐只读取已发布且未过期的活动。
- 后续可以接入 LLM 做活动内容初审，例如检测夸大收益、敏感内容、信息缺失，但最终由人工确认。

面试表达：这不是单纯 CRUD，而是把用户生成内容引入状态机、权限控制和风控审核，保证推荐系统的数据源可信。

## 10. 推荐反馈闭环

问题：当前推荐系统主要依赖离线画像、检索分数和 LLM 理由，用户是否真的满意还没有反向进入系统。推荐如果没有反馈闭环，后续很难越用越准。

优化方向：

- 在推荐结果上增加反馈按钮：`感兴趣`、`不感兴趣`、`已预约`、`不相关`。
- 把点击、预约、完成、取消、忽略等行为记录为推荐反馈事件。
- 用户画像异步抽取时读取这些反馈，调整兴趣标签、地点偏好、收益偏好。
- 评价集之外再增加在线指标：推荐点击率、预约转化率、完成率、负反馈率。
- 后续可以做多版本推荐策略 A/B 对比。

面试表达：从一次性推荐升级为“推荐 -> 反馈 -> 画像更新 -> 下次推荐变准”的闭环系统。

## 11. Outbox / MQ 运维面板

问题：项目已经有 Outbox Pattern 和失败重试表，但目前主要停留在后端表结构和定时投递逻辑。开发者需要查数据库才能知道哪些消息失败、失败原因是什么。

优化方向：

- 增加管理端页面查看 `domain_event_outbox`。
- 支持按状态筛选：`PENDING`、`FAILED`、`SENT`。
- 展示 routingKey、payloadType、attempts、nextAttemptAt、lastError。
- 对失败消息提供手动重试按钮。
- 对超过最大重试次数的消息做告警或人工处理。

面试表达：这能把“消息最终一致性”从代码机制补成可观测、可运维的工程闭环。

## 12. Agent 观测面板

问题：项目已经记录 Agent Run / Step 日志，但用户和开发者还不能直观看到一次 AI 推荐、计划生成或 Coach 对话到底经过了哪些步骤。

优化方向：

- 做一个 Agent Trace 页面，展示每次运行的 runId、用户输入、步骤状态、耗时、是否 fallback。
- 每个步骤展示摘要：Query Rewrite、MCP Context、Retrieval、LLM Recommendation、Rule Fallback、Response Build。
- 展示 token 消耗、模型名称、失败原因和最终输出。
- 支持按用户、时间、功能类型筛选。
- 后续可以把 badcase 直接从 trace 转成评价集 case。

面试表达：Agent 项目不能只看最终回答，还要能观察中间状态，否则很难定位是 query 理解错、召回错、rerank 错还是模型输出错。

## 13. 权限和风控

问题：当前项目已经有注册登录，但角色权限还可以继续细化。学生端、社会端、管理员端如果权限边界不清晰，真实生产里会出现越权访问、恶意发布、刷接口等问题。

优化方向：

- 引入角色：`STUDENT`、`ORGANIZATION`、`ADMIN`。
- 社会端只能管理自己发布的活动。
- 管理员拥有审核、下架、查看 Outbox/Agent 日志等权限。
- 对高频接口加限流，例如 AI 推荐、计划生成、活动发布。
- 对活动内容、Coach 输入、推荐反馈做基础风控记录。

面试表达：这部分可以讲认证、授权、接口限流、数据隔离和 AI 功能滥用防护。

## 14. 推荐质量持续评测

问题：当前已有 `retrieval-eval-set.json` 和 A/B 评测接口，但评价集规模还小，指标主要关注 TopK 命中和 MRR，更多是在防止检索退化，还不够像完整推荐系统评测。

优化方向：

- 扩充评价集，覆盖日语、编程、研究、线上兼职、报酬、地点约束、多轮对话等场景。
- 从二值相关升级为多级相关，例如 `0=不相关`、`1=弱相关`、`2=相关`、`3=强相关`。
- 增加 `NDCG@K`、`Recall@K`、`MRR`、零结果率、过期活动返回率、无效 eventId 率。
- 把线上 badcase 一键沉淀为离线评测 case。
- 每次调整 Query Rewrite、召回权重、rerank prompt 后跑一遍评测，记录 current vs baseline。

面试表达：这能说明项目不是靠感觉调 prompt，而是有离线评价集、A/B 对比和可复现指标支撑推荐优化。
