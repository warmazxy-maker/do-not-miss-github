# 能力评分公平性与防刷分 v2

## 目标

这次优化不是让 LLM 变成“更聪明的裁判”，而是给评分链路增加服务端强制约束：

```text
完成活动或挑战
-> 构造证据快照
-> 重复证据检查
-> Evidence Assessment Agent
-> 团队贡献约束
-> 多能力维度预算
-> Java 确定性评分
-> 安全写入或回放复核
-> 用户申诉
```

LLM 仍然只负责把非结构化经历提取为难度、完成质量、个人贡献、能力维度和证据引用。是否重复、最多能给多少贡献、多个维度如何共享收益、旧分数如何替换，全部由后端确定性逻辑决定。

## 1. 重复证据指纹

系统在原有 `evidence_hash` 之外增加 `content_fingerprint`。

- `evidence_hash` 包含记录 ID、时间等完整快照，用于判断一次评估是否完全相同。
- `content_fingerprint` 只保留来源类型、标题、组织、分类、内容、技能、`did` 和 `learned` 等实质内容，并统一大小写、空格和标点。
- 指纹使用 SHA-256 生成。

当同一用户用不同成就记录提交相同实质内容时：

```text
fairness_status = DUPLICATE_REVIEW
status = REVIEW_REQUIRED
duplicate_of_job_id = 原任务 ID
```

该任务不会继续调用 LLM，也不会再次进入 Java 评分，因此不会重复增加能力值。

当前实现识别的是规范化后完全相同的内容。近义改写、同一项目拆成多段描述等语义级重复，后续可以增加 embedding 相似度和时间窗口检测。

## 2. 团队贡献上限

团队经历不能仅凭用户自述获得接近满分的个人贡献。

识别到团队、小组、协作、合作、`team` 或 `group` 等语境后：

| 情况 | 个人贡献上限 | 贡献置信度上限 |
|---|---:|---:|
| 没有清楚描述本人负责内容 | 0.55 | 0.45 |
| 已描述本人职责，但没有独立验证 | 0.85 | 0.75 |

被限制时会写入：

```text
FAIRNESS_CONTRIBUTION_CAPPED
```

公司或组织背景仍可以带来较高的来源可信先验，但不能替代个人贡献证据。高质量个人项目也可以凭公开代码、可运行 Demo、测试和明确职责获得更高评价。

## 3. 多能力维度预算

一个活动最多可以提取五个能力维度。如果每个维度都以接近 1.0 的相关度独立计分，用户可以通过“多贴标签”放大总收益。

后端为一次经历设置固定相关度预算：

```text
total relevance budget = 2.0
```

当所有有效维度相关度之和超过 2.0 时，按同一比例缩放：

```text
multiplier = 2.0 / sum(original relevance)
adjusted relevance = original relevance * multiplier
```

例如三个维度均为 `0.9`：

```text
原始总和 = 2.7
缩放倍率 = 0.7407
调整后 = 0.6667 / 0.6667 / 0.6667
```

原始相关度、调整后相关度和预算倍率都会进入 `factor_snapshot_json`，方便解释和审计。

## 4. 证据修改后的安全重算

用户补写复盘后，证据内容可能变化。系统不能在旧结果之上再加一次分，也不能直接覆盖历史。

### 可以直接替换

如果待修正结果仍是该能力时间线上的最新有效结果：

1. 从旧结果评分前的能力状态重新计算。
2. 创建新的评分结果。
3. 新结果写入 `supersedes_result_id`。
4. 旧结果改为 `SUPERSEDED`。
5. 当前能力状态替换为新结果。

这样实现的是“修正同一次经历”，而不是“再奖励一次经历”。

### 必须回放

如果旧结果之后已经存在其他活动的评分，直接替换会破坏时间顺序。系统不会冒险覆盖当前能力状态，而是创建：

```text
request_type = EVIDENCE_CHANGED_REPLAY
status = PENDING
```

接口响应中的 `correctionPending=true` 表示需要按时间顺序重放该能力的评分历史。当前阶段已建立待处理记录和安全阻断，自动回放执行器属于后续增强。

## 5. 用户申诉

用户可以对自己的一条评分结果提交申诉：

```http
POST /api/ability-scoring/results/{resultId}/appeals
Authorization: Bearer <token>
Content-Type: application/json

{
  "reason": "个人贡献证据没有被完整识别",
  "evidenceNote": "我负责了接口设计、测试与部署，相关材料见复盘记录"
}
```

查看申诉：

```http
GET /api/ability-scoring/appeals
Authorization: Bearer <token>
```

同一评分存在 `PENDING` 或 `IN_REVIEW` 申诉时，系统拒绝重复提交。申诉和证据修改回放统一保存在 `ability_score_appeals`，但通过 `request_type` 区分。
