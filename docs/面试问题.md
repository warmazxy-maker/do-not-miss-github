# do not miss 面试问题

## 一、项目整体

### 1. 你这个项目是做什么的？

答题思路：

这个项目是一个大学生成长机会平台。社会端可以发布实践事件，学生端可以搜索、AI 推荐、预约和扫码完成。完成后的事件会进入个人成就模块，学生可以记录反思，系统基于历史经历生成统计图、成长曲线和智能简历。后来我还加入了挑战模块，让学生可以自定义目标，比如学完一个项目、减肥、英语过六级，完成后也会沉淀到成就系统里。

重点：

- 外部事件
- 个人挑战
- 预约签到
- 成就沉淀
- AI 推荐和智能简历

### 2. 这个项目和普通活动报名系统有什么区别？

普通活动报名系统只解决“报名参加活动”。

这个项目多了三点：

- AI 推荐：根据学生需求和历史经历推荐事件。
- 成长沉淀：事件和挑战完成后进入个人成就。
- 智能简历：把经历转化为可展示的成长档案。

可以总结：

> 普通系统关注活动本身，我这个项目更关注学生成长路径。

### 3. 为什么要加挑战模块？

因为纯事件平台有冷启动问题。如果没有组织发布活动，学生端就没内容。挑战模块允许学生自己创建目标，即使没有外部活动，也能积累成长记录。

进一步价值：

- 让平台早期可用。
- 丰富学生画像。
- 给 AI 推荐提供更多上下文。
- 和事件共同进入个人成就。

### 4. 你项目的核心业务闭环是什么？

可以这样回答：

> 社会端发布事件，学生端搜索或 AI 推荐事件，学生预约后通过扫码完成，完成后进入个人成就；学生也可以创建个人挑战，完成挑战后同样进入个人成就；成就记录再反过来用于 AI 推荐和智能简历生成。

## 二、Spring Boot 和 Java

### 5. 你为什么选 Spring Boot？

因为 Spring Boot 适合快速构建后端服务，生态成熟，集成 Web、JPA、Redis、校验、配置管理都比较方便。

项目里用到：

- `spring-boot-starter-web`
- `spring-boot-starter-data-jpa`
- `spring-boot-starter-data-redis`
- `spring-boot-starter-validation`
- Flyway

### 6. Controller、Service、Repository 分别负责什么？

Controller：

- 接收 HTTP 请求
- 参数校验
- 调用 Service
- 返回 DTO

Service：

- 处理业务逻辑
- 控制事务
- 调用多个 Repository 或其他 Service

Repository：

- 数据库访问
- 基于 JPA 操作实体

不要把业务逻辑写在 Controller 里，因为 Controller 应该薄，业务应该集中在 Service。

### 7. 为什么要用 DTO，不直接返回 Entity？

原因：

- 避免暴露数据库结构。
- 可以控制返回字段。
- 防止懒加载序列化问题。
- 请求和响应模型可以独立变化。
- Entity 改动不会直接影响前端。

比如 `EventEntity` 是数据库模型，`EventResponse` 是接口响应模型。

### 8. 你项目中哪里用了事务？

典型场景：

- 发布事件时自动创建组织。
- 预约事件。
- 扫码完成时更新预约并创建成就。
- 完成挑战时更新挑战并创建成就。
- 更新历史反思。

重点回答：

> 完成挑战和扫码完成必须用事务，否则可能出现状态更新成功但成就记录创建失败的数据不一致。

### 9. `@Transactional(readOnly = true)` 有什么作用？

用于只读查询方法。

好处：

- 表达语义：这个方法不会修改数据。
- 一些数据库和 ORM 可以做优化。
- 防止误操作。

项目里查询事件、历史记录、统计数据时可以使用。

### 10. Java 里为什么枚举适合表示状态？

比如：

- `ReservationStatus`
- `ChallengeStatus`
- `BenefitType`
- `UserRole`

枚举优点：

- 类型安全。
- 可读性好。
- 避免魔法字符串。
- 状态范围固定。

## 三、登录认证

### 11. 注册登录是怎么做的？

注册：

1. 校验用户名、邮箱、密码。
2. 判断用户名和邮箱是否重复。
3. 生成 `publicUserId`。
4. 密码 BCrypt 哈希。
5. 保存到 `users` 表。
6. 生成登录 token。
7. token 存 Redis。

登录：

1. 根据用户名或邮箱查用户。
2. BCrypt 校验密码。
3. 生成 token。
4. token 存 Redis。
5. 返回 token 和用户信息。

### 12. 为什么密码不能明文存？

明文存储风险极高，一旦数据库泄露，用户密码直接暴露。

应该使用哈希算法，比如 BCrypt。

BCrypt 特点：

- 自动加盐。
- 计算成本可调。
- 抗彩虹表。

### 13. 为什么 token 存 Redis？

原因：

- 可以设置过期时间。
- 可以主动退出登录。
- 可以服务端强制 token 失效。
- 查询速度快。

和 JWT 对比：

- Redis token 是有状态的。
- JWT 更适合无状态服务。
- Redis token 更方便注销。
- JWT 撤销麻烦，需要黑名单或短过期时间。

### 14. 前端请求怎么识别用户？

请求头带：

```http
Authorization: Bearer <token>
```

后端 `CurrentUser` 从 header 中解析 token，再去 Redis 查对应的 `publicUserId`。

### 15. 为什么用 `publicUserId` 而不是数据库自增 ID？

自增 ID 容易暴露用户数量和内部结构。

`publicUserId` 更适合作为业务层用户标识。

另外项目早期有 demo 数据，比如 `demo-student`，用 `publicUserId` 可以兼容这些已有数据。

## 四、MySQL

### 16. 这个项目有哪些核心表？

核心表：

- `users`
- `organizations`
- `events`
- `follows`
- `reservations`
- `challenges`
- `achievement_records`

### 17. 为什么成就记录要存快照？

因为事件或挑战以后可能被修改，但学生完成时的经历应该保持历史真实性。

例如：

- 活动标题修改了
- 组织名称修改了
- 挑战描述修改了

历史成就不应该被影响。

所以 `achievement_records` 会保存标题、组织、分类、内容、技能等快照字段。

### 18. 关注表为什么需要唯一索引？

```sql
UNIQUE (user_id, organization_name)
```

防止同一用户重复关注同一组织。

这个约束不能只依赖前端，因为前端可能重复提交，或者并发请求同时到达。

### 19. 哪些字段需要索引？

事件表：

- 分类
- 时间
- 组织

预约表：

- 用户 + 状态
- 二维码 token

挑战表：

- 用户 + 状态
- 用户 + 创建时间

成就表：

- 用户 + 完成时间
- 用户 + 来源类型 + 来源 ID

### 20. 如果数据量很大，普通搜索怎么办？

当前是 MySQL `LIKE` 搜索，适合原型和小数据量。

数据量变大后可以：

- 接 Elasticsearch / OpenSearch。
- 对事件标题、内容、技能做全文索引。
- 把搜索和推荐分离。
- 对热门搜索做缓存。

## 五、Redis

### 21. Redis 在项目里用来做什么？

两个用途：

- 登录 token
- 二维码 token

### 22. 为什么二维码 token 适合放 Redis？

二维码 token 是短期数据，Redis 支持 TTL。

扫码完成后直接删除 token，可以防止重复使用。

### 23. Redis 挂了怎么办？

影响：

- 登录态无法校验。
- 扫码 token 无法校验。

解决思路：

- Redis 高可用。
- 登录 token 可以考虑短时间降级。
- 二维码 token 可以数据库兜底，但安全性要权衡。

项目里目前数据库保存了 `qrToken`，Redis 过期后可以兜底查数据库。

### 24. Redis token 和数据库 session 哪个好？

Redis 更适合 session：

- 读写快。
- TTL 方便。
- 不给 MySQL 增加高频访问压力。

数据库更适合持久业务数据。

## 六、预约扫码

### 25. 预约流程怎么设计的？

1. 学生选择事件。
2. 创建预约记录。
3. 生成二维码 token。
4. token 存 Redis。
5. 返回给前端。

### 26. 扫码完成怎么保证数据一致？

扫码完成时需要事务：

1. 校验 token。
2. 查询预约。
3. 修改预约状态为 `COMPLETED`。
4. 删除 Redis token。
5. 创建成就记录。

如果其中任一步失败，应该回滚数据库操作。

### 27. 怎么防止重复扫码？

可以从几个层面做：

- 扫码成功后删除 Redis token。
- 预约状态改成 `COMPLETED`。
- 成就记录按 `user_id + source_type + source_id` 防重复。
- 再次扫码时检查状态。

### 28. 真实线下场景如何防代签？

可以增加：

- 活动时间窗口校验。
- 地理位置校验。
- 动态二维码，短时间刷新。
- 组织端确认。
- 人脸或学生证校验。

## 七、挑战模块

### 29. 挑战模块解决什么问题？

解决平台早期事件少、学生仍然需要自我成长记录的问题。

挑战是学生自定义目标，例如：

- 学完一个项目
- 英语过六级
- 减肥到 70kg
- 完成作品集

### 30. 挑战和事件有什么区别？

事件：

- 外部组织发布
- 有时间地点
- 需要预约和扫码

挑战：

- 学生自己创建
- 目标导向
- 完成时填写反思

共同点：

- 完成后都进入个人成就。
- 都参与统计和 AI 分析。

### 31. 为什么挑战完成后要进入成就模块？

因为挑战也是成长经历。

如果学生完成了“学完黑马商城项目”，这对求职很有价值，应该进入智能简历和成长曲线。

## 八、个人成就和统计

### 32. 个人成就模块有什么用？

它把学生完成的事件和挑战沉淀为历史档案。

包括：

- 历史记录
- 做了什么
- 学到了什么
- 类别统计
- 成长曲线
- 智能简历

### 33. 类别统计怎么做？

按成就记录里的 `category` 分组统计数量。

事件类别可能是：

- 公益
- 企业
- 研究

挑战类别可能是：

- 技能挑战
- 考试挑战
- 身体挑战

### 34. 成长曲线怎么做？

后端根据历史记录文本对能力维度打分，例如：

- 日语、沟通、翻译 -> 沟通表达
- 活动、现场、团队 -> 执行协作
- 调研、访谈、问卷 -> 调研分析
- 内容、文案、记录 -> 内容创作

每完成一条记录，累计对应能力分数，形成成长曲线。

### 35. 为什么统计图不直接交给 AI？

因为统计图需要准确、可复现、可解释。

AI 可能会幻觉或生成不稳定数字。

所以：

- 后端负责计算真实数据。
- 前端负责画图。
- AI 负责解释数据。

## 九、AI 相关

### 36. AI 推荐事件怎么做？

输入：

- 学生需求
- 候选事件
- 学生历史成就

输出：

- 推荐事件
- 推荐分数
- 推荐理由

当前 mock 规则：

- 关键词匹配
- 需求意图匹配
- 历史经历相关性

### 37. 如果接真实大模型，怎么改？

保持接口不变，只替换 `AiService` 内部实现。

流程：

1. 后端查询候选事件。
2. 后端查询学生历史。
3. 拼 prompt。
4. 请求大模型。
5. 解析推荐结果。
6. 返回给前端。

### 38. AI 推荐会不会很慢？

可能会。

优化方式：

- 先用规则召回候选事件，减少输入数量。
- 对学生画像做缓存。
- 对相同需求做短期缓存。
- 异步生成解释。
- 设置超时和降级策略。

降级：

> 如果 AI 超时，就返回普通搜索结果加简单规则推荐。

### 39. 怎么防止 AI 幻觉？

做法：

- 只允许 AI 从候选事件中选择。
- 返回事件 ID，而不是让 AI 自己编事件。
- 后端校验 AI 返回的 ID 是否存在。
- 统计数字由后端计算，不让 AI 生成。
- prompt 中明确禁止编造经历。

### 40. 智能简历怎么保证可信？

只基于用户真实完成记录和反思生成。

不能让 AI 编造经历。

输出可以包括：

- 简历总结
- bullet points
- 能力关键词
- 下一步建议

## 十、分页和性能

### 41. 为什么要分页？

挑战和历史记录会不断增长，如果一次性展示全部：

- 前端页面会越来越长。
- 请求数据量变大。
- 用户体验差。

所以前端和后端都支持分页。

### 42. 分页接口怎么设计？

接口：

```http
GET /api/challenges/page?page=0&size=5
GET /api/achievements/history/page?page=0&size=5
```

返回：

- `content`
- `page`
- `size`
- `totalElements`
- `totalPages`
- `first`
- `last`

### 43. page 从 0 开始还是 1 开始？

后端通常用 Spring Data 的 0-based page。

前端展示可以显示成 1-based，比如 `1 / 3`。

## 十一、安全

### 44. 当前项目安全性还有哪些不足？

可以主动承认：

- 没有接 Spring Security 完整过滤链。
- 没有 refresh token。
- 没有验证码和登录失败限流。
- demo 用户有明文兼容逻辑，只适合演示。
- 前端原型本地存储 token，生产环境要考虑 XSS 风险。

### 45. 如果生产化，认证会怎么升级？

可以升级：

- Spring Security 过滤器。
- JWT + refresh token。
- Redis token 黑名单。
- 登录失败次数限制。
- HTTPS。
- 密码复杂度和验证码。
- 管理端权限控制。

## 十二、并发和一致性

### 46. 如果用户同时点两次完成挑战，会怎样？

理想处理：

- 数据库唯一约束防止重复成就。
- Service 层先检查成就是否存在。
- 事务保证状态和成就一致。

当前项目 Service 里用 `sourceType + sourceId` 做幂等判断。

### 47. 如果两个请求同时预约同一个事件呢？

当前一个学生预约同一个事件有唯一约束：

```sql
UNIQUE (user_id, event_id)
```

如果并发插入，数据库会阻止重复。

如果未来事件有人数限制，需要加库存扣减逻辑：

- 乐观锁
- 悲观锁
- Redis 预扣库存

## 十三、可扩展性

### 48. 如果事件很多，推荐系统怎么升级？

可以分三层：

1. 召回：搜索引擎或向量检索拿候选事件。
2. 粗排：规则或轻量模型排序。
3. 精排：大模型结合学生画像生成理由。

### 49. 如果要做组织端权限，怎么设计？

新增：

- 组织表和用户组织关系表。
- 用户角色：学生、组织成员、组织管理员、平台管理员。
- 发布事件时校验用户是否属于该组织。
- 我的事件只查当前组织发布的事件。

### 50. 如果要把前端从原生 JS 改成 Vue/React，后端要改吗？

不需要大改。

只要后端 API 稳定，前端技术栈可以替换。

这就是前后端分离的好处。

## 十四、高频追问速答

### 51. 你项目最大的亮点是什么？

不是单纯活动报名，而是事件 + 挑战 + 成就 + AI 的成长闭环。

### 52. 你项目最难的点是什么？

难点不在 CRUD，而在如何把不同来源的成长经历统一抽象成成就记录，并让 AI 在可信数据基础上做推荐和总结。

### 53. 为什么 Redis 不存成就记录？

成就记录是核心持久数据，需要可靠存储和查询分析，应该放 MySQL。Redis 只放临时 token。

### 54. 为什么不一开始就接真实 AI？

因为应该先稳定业务输入输出。当前 mock AI 保证没有 API Key 也能跑通完整流程。后续替换 `AiService` 即可。

### 55. 如果让你继续优化，你会做什么？

优先级：

1. 接真实大模型。
2. 前端接后端 API。
3. 用 Spring Security 完善认证授权。
4. 加 Elasticsearch 或向量检索。
5. 增加组织端审核和活动人数限制。
6. 加测试和 CI。
