# Do Not Miss 待处理问题

这份文档只记录尚未实施或尚未定稿的后续优化方向。已经落地的问题应迁移到后端 `PROBLEM_LOG.md`，不要继续留在这里。

## 001. AI 推荐链路与 AI 规划链路需要升级为 LLM 主导的候选筛选与重排

状态：待设计。

### 当前情况

当前 AI 推荐链路大致是：

1. 用户输入自然语言需求。
2. Query Rewrite 将需求整理成结构化查询信号。
3. RAG / OpenSearch 召回活动候选。
4. Java 本地打分引擎融合关键词、BM25、语义分、意图分、用户画像分、元数据分等维度。
5. 分数靠前的候选活动交给 LLM。
6. LLM 在候选集内选择推荐结果，并生成推荐理由。

AI 规划功能也依赖类似候选来源：先召回可用活动，再由 Planner / Schedule Checker / Critic 等 Agent 生成计划。

### 发现的问题

当前链路的工程可控性比较强，但部分打分维度开始显得僵硬：

- 意图识别分依赖人工规则和关键词表，容易把需求理解压扁。
- 用户画像分主要是规则匹配，不能很好理解用户真实阶段、成长路径和长期目标。
- 本地打分引擎权重越来越复杂，后续调参成本会变高。
- LLM 已经具备较强的候选理解能力，但当前只能看本地打分后的 TopN，参与空间有限。
- 企业端上传活动时已经生成了活动安全质量报告，但推荐链路尚未充分利用这份结构化质量信息。

### 优化方案

将推荐链路从“本地打分主导 + LLM 解释”升级为“RAG 候选筛选 + LLM 适配性评价与重排”。

新的推荐链路建议：

1. 用户输入自然语言需求。
2. Query Rewrite 继续负责提取查询意图、约束和上下文。
3. RAG 负责做语义召回和关键词召回，只做候选筛选，不再承担复杂规则评分。
4. RAG 最多返回 15 个候选活动。
5. LLM 同时读取：
   - 候选活动基础信息。
   - 活动安全质量报告。
   - 用户画像。
   - 用户个人成长标签、能力等级和成绩。
   - 当前多轮对话上下文。
6. LLM 判断每个候选是否适合该用户，并输出：
   - 推荐 / 不推荐。
   - 排序。
   - 匹配理由。
   - 不合适理由。
   - 风险提示。
7. 后端只做硬约束校验，例如 eventId 必须来自候选集、过期活动不能推荐、未通过审核活动不能推荐。

### 预期效果

- 本地规则从“判分器”退回到“召回过滤与安全校验器”。
- LLM 能真正参与“适不适合这个用户”的判断。
- 活动质量报告、用户能力标签、画像和历史成绩可以进入推荐决策。
- 推荐理由更贴近用户成长路径，而不是只解释关键词命中。
- AI 规划功能后续也可以复用同一套候选筛选接口。

### 面试表达

早期推荐链路使用本地规则打分保证可控性，但随着用户画像、活动质量报告和能力评分体系变复杂，纯规则权重开始僵硬。后续我计划把本地打分引擎弱化为召回与校验层，由 RAG 提供最多 15 个高质量候选，再让 LLM 结合活动质量报告、用户画像、成长标签和能力等级进行适配性重排。这样既保留了 RAG 的可控候选边界，又让模型承担更接近真实推荐判断的部分。

## 002. AI 推荐需要升级为 Agentic RAG 引导者

状态：待设计。

### 当前情况

当前 AI 推荐更接近一次性请求：

1. 用户提交需求。
2. 后端完成 Query Rewrite、RAG、LLM 推荐。
3. 前端展示推荐结果。

虽然已有多轮 Query Rewrite 和用户画像，但整体仍偏“伪记忆”：系统能合并上一轮需求，但还没有一个真正面向用户持续对话的 Agent。

### 发现的问题

- 用户可能不是一次就说清楚需求，而是边看推荐结果边补充条件。
- 当前推荐结果返回后，用户如果继续追问，系统很难围绕上一轮候选继续交流。
- 每次请求都容易重新触发完整 RAG，成本和延迟较高。
- 推荐 Agent 没有明确的“是否调用 RAG”的工具决策能力。
- 用户画像、成长标签和能力等级虽然存在，但还没有成为推荐对话 Agent 的常驻记忆。

### 优化方案

新增一个面向用户的推荐对话 Agent，名称暂定为“引导者”。

引导者的职责：

- 读取用户输入并判断用户真实意图。
- 决定是否调用 RAG。
- 如果需要推荐活动，则调用原有 RAG 链路。
- 如果用户只是围绕已有结果继续聊天，则不触发 RAG。
- 保存短期对话上下文。
- 当上下文接近 token 上限时进行记忆压缩。
- 常驻读取用户画像、成长标签、能力等级和历史偏好。

建议调用链路：

```text
用户输入
  -> 引导者 LLM 判断意图
      -> 需要新推荐：调用 RAG 推荐链路
          -> RAG 返回候选和推荐结果
          -> 推荐结果同时返回给引导者和前端
          -> 引导者可继续围绕这些结果与用户对话
      -> 不需要新推荐：基于短期上下文、常驻记忆和已有推荐结果直接回复
```

第一阶段只做 AI 推荐的引导者。AI 规划模块如何升级为 Agentic Planning 另行设计。

### 记忆设计

短期记忆：

- 当前会话用户需求。
- 最近推荐过的活动。
- 用户对推荐结果的追问、否定和补充条件。
- 最近一次 RAG 结果摘要。

压缩策略：

- token 使用超过阈值时，将旧对话压缩成结构化摘要。
- 摘要保留需求变化、已否定条件、已接受偏好和推荐过的活动。
- token 阈值暂不定，后续通过压测和真实对话长度决定。

常驻记忆：

- 用户画像。
- 用户能力标签。
- 能力等级和可信度。
- 历史完成活动。
- 当前挑战和成长方向。

### 预期效果

- AI 推荐从“一次性接口”升级为“可连续引导的 Agent”。
- 用户可以自然追加条件，例如“最好线上”“不要太难”“有没有更偏 Java 的”。
- 系统可以减少无意义 RAG 调用。
- 推荐结果可以被继续讨论、筛选和解释。
- 为后续 AI 规划的多轮 Agent 化打基础。

### 面试表达

我计划把推荐接口升级成 Agentic RAG：用户不再只是提交一次查询，而是和一个“引导者 Agent”持续对话。引导者会基于用户画像和短期上下文判断是否需要调用 RAG，如果只是围绕已有推荐结果追问，就直接基于上下文回答；如果用户提出新目标，再调用 RAG。这样能把推荐从单次检索变成多轮交互式推荐，同时通过记忆压缩控制 token 成本。

## 003. 新用户简历定段与面试官 Agent

状态：待研究。

### 当前情况

当前能力评分主要来自平台内部行为：

- 用户完成活动。
- 用户完成挑战。
- 用户写复盘日志。
- 系统根据活动证据、用户复盘、质量报告和评分 Skill 更新能力分。

这套系统适合老用户，但新用户刚注册时缺少平台内行为数据，冷启动能力画像较弱。

### 发现的问题

- 新用户没有完成记录，系统难以判断初始能力。
- 如果只靠用户自填标签，很容易虚高或失真。
- 外部简历里有大量能力证据，但当前系统还不会读取和评估。
- 简历评价和活动加分必须遵循同一套评分认知，否则“简历定段的一分”和“完成活动加的一分”价值不一致。
- 不同行业、岗位和面试官风格不同，不能简单给每个行业都微调一个独立模型。

### 优化方案

新增“面试官 Agent”，用于新用户简历冷启动定段。

触发方式：

- 新用户注册后，系统询问是否愿意上传简历进行能力评估。
- 简历越完整，评估越可靠。
- 用户可选择跳过，后续仍然通过平台活动逐步积累能力。

面试官 Agent 的核心能力：

1. 读取外部简历。
2. 抽取教育、项目、实习、比赛、证书、技能等证据。
3. 持有与成长标签加分 Agent 一致的评分 Skill / Rubric。
4. 根据岗位或行业知识库生成追问。
5. 通过多轮问答验证简历真实性和能力深度。
6. 在短期记忆中持续保存问答上下文和能力判断。
7. 最终输出能力证据，而不是直接修改最终分数。
8. 统一交给 AbilityScoreEngine 计算初始能力分。

建议数据流：

```text
用户上传简历
  -> Resume Parser Agent 抽取结构化经历
  -> 面试官 Agent 读取行业知识库和统一评分 Skill
  -> 生成第一轮追问
  -> 用户回答
  -> Answer Evaluator 评估回答质量、可信度、不确定度
  -> 必要时继续追问
  -> 输出 AbilityEvidence
  -> AbilityScoreEngine 定段
```

### 行业知识库问题

当前待研究问题：

- 行业知识库应该主要来自牛客、小红书等真实面经，还是来自八股知识和岗位能力标准？
- 真实面经更接近面试现场，但噪声大、版权和质量不稳定。
- 八股知识更稳定，但容易变成背题系统，不一定反映真实岗位能力。

初步建议：

- 第一版只做 AI 后端 / Agent 工程实习方向。
- 知识库分成两类：
  - 岗位 Rubric：稳定能力标准，例如 RAG、Agent、Java、MQ、MySQL、Redis。
  - 面试问题库：真实追问风格和常见问题。
- 面经可以作为问题风格和追问样本，但不要直接作为唯一评分标准。

### 微调问题

当前待研究问题：

- 如果要通过微调提升面试官判分能力，应该如何构建训练数据？
- 不同行业判分标准不同，如何避免给每个行业单独微调？
- 如何把“判分能力”和“行业知识”解耦？

初步判断：

- 不建议微调行业知识。
- 行业知识通过 RAG / Rubric / 配置注入。
- 微调重点应该放在通用评价能力上：
  - 证据是否能证明某项能力。
  - 回答是否具体。
  - 是否有工程细节。
  - 是否和简历一致。
  - 置信度和不确定度如何判断。
  - 是否需要继续追问。

核心原则：

```text
RAG 负责给标准，微调负责学判断。
```

### 预期效果

- 新用户可以通过简历和面试问答获得初始能力画像。
- 简历定段和活动加分共享同一套评分认知。
- 高分但不确定的能力可以继续进入 Judge 验证。
- 系统从“记录成长”扩展到“评估已有能力 + 引导后续成长”。

### 面试表达

为了处理新用户冷启动，我计划引入面试官 Agent。它会读取用户简历，结合岗位知识库和统一评分 Rubric 进行追问，并把用户回答转成能力证据，再交给统一 AbilityScoreEngine 定段。这里我不会让 LLM 直接拍脑袋给分，而是把简历、面试回答、活动完成记录都抽象为 AbilityEvidence，保证不同证据来源进入同一套评分体系。后续如果做微调，也主要微调模型的证据评价能力，而不是把行业知识写死在模型参数里。

### 最终版解决方案补充：定分数与涨分数两条线

该优化最终应拆成两条业务链路：

- 定分数：新用户刚进入平台，通过简历和面试问答判断初始能力水平。
- 涨分数：老用户完成活动、挑战或复盘后，判断这次经历能带来多少成长增量。

两条线的输入来源不同，但不能各自使用一套评分价值观。最终都必须走同一个核心评分闭环：

```text
不同来源的事实证据
  -> 统一 Evidence Evaluator Agent
  -> Java AbilityScoreEngine
  -> UserAbilityState / AbilityScoreResult
  -> 必要时进入 Judge Agent
```

#### 活动 / 挑战完成后的涨分线

这条线对应平台内发生的新成长事件：

```text
用户完成活动或挑战
  -> Achievement Recorder 保存完成记录
  -> Evidence Snapshot Builder 构建标准证据快照
  -> Evidence Evaluator Agent 评价证据质量和能力维度
  -> Java AbilityScoreEngine 计算经验增量、能力分增量和不确定度变化
  -> Judge Agent 在高风险或高影响场景下触发验证
```

其中真正承担 LLM 语义判断的是 Evidence Evaluator Agent。它不直接输出最终加分，而是输出评分原料：

```json
{
  "sourceType": "ACTIVITY_COMPLETION",
  "activityDifficulty": 65,
  "completionQuality": 0.78,
  "personalContribution": 0.72,
  "assessmentConfidence": 0.68,
  "dimensions": [
    {
      "dimension": "Java 后端开发",
      "normalizedDimension": "java-backend",
      "relevance": 0.86,
      "confidence": 0.72
    }
  ],
  "evidenceFindings": [],
  "riskFlags": []
}
```

Java AbilityScoreEngine 只认这类统一结构，然后按固定公式计算分数。这样可以避免 LLM 直接决定“加几分”。

#### 新用户简历 / 面试定分线

这条线对应外部经历冷启动：

```text
用户上传简历
  -> Resume Parser Agent 抽取教育、项目、实习、比赛、证书和技能声明
  -> Interviewer Agent 根据简历、岗位 Rubric 和行业知识库进行追问
  -> 用户回答面试问题
  -> Evidence Evaluator Agent 将简历片段、问题、回答和上下文转成统一证据结构
  -> Java AbilityScoreEngine 计算初始能力分
  -> Judge Agent 对高分但不确定的能力触发验证
```

第一版不建议拆太多 Agent。推荐先做 3 个 Agent：

1. Resume Parser Agent：只负责简历结构化，不评分。
2. Interviewer Agent：第一版同时承担面试规划和追问，不直接判最终分。
3. Evidence Evaluator Agent：和活动涨分线共用，持有统一评分 Skill / Rubric，负责评价证据质量。

后续复杂度上来后，再把 Interviewer Agent 拆成：

- Interview Planner Agent：决定优先验证哪些能力点。
- Interviewer Agent：负责具体多轮对话和追问。
- Answer Evaluator Agent：可作为 Evidence Evaluator 的面试场景适配层。

#### 统一评分价值观

两条线必须共享：

- 同一份评分 Skill / Rubric。
- 同一套 Evidence Schema。
- 同一个 Evidence Evaluator Agent。
- 同一个 Java AbilityScoreEngine。
- 同一套 Judge 触发规则。
- 同一套评分审计和版本记录。

核心原则：

```text
活动 Agent、面试官 Agent、简历解析 Agent 都不能直接改最终分。
它们只能收集或组织证据。
所有证据必须先进入统一 Evidence Evaluator，再由 Java AbilityScoreEngine 计算。
```

这样才能保证：

- 做完活动加的 1 分，和简历定段得到的 1 分价值一致。
- 不同 Agent 不会各自为政。
- 后续微调时只需要优化 Evidence Evaluator 的证据评价能力，而不是给每个行业单独微调一套评分模型。
- 行业知识通过 RAG / Rubric 注入，模型微调重点放在通用判分能力上。

## 004. 模型推理优化与本地推理服务

状态：学习规划中。

### 当前情况

当前项目主要调用远程大模型 API，例如 Qwen 系列模型。后端已经有统一 LLM 客户端、模型配置和部分任务级调用逻辑，但还没有真正进入本地推理服务和推理引擎优化。

### 发现的问题

- 远程 API 的延迟、稳定性和成本不可完全控制。
- 不同任务复杂度差异很大，但当前模型路由还不够精细。
- 同步接口使用大模型会明显影响用户等待时间。
- 如果未来用户量增加，只靠远程 API 难以评估成本上限。
- 推理引擎方向目前还没有实践积累。

### 优化方案

后续引入本地推理服务实验线，优先研究 vLLM。

第一阶段目标不是从零写推理引擎，而是：

- 使用 vLLM 部署一个本地开源模型。
- 后端支持把部分 AI 任务路由到本地 vLLM。
- 记录 TTFT、TPOT、tokens/s、p95 延迟和显存占用。
- 对比不同模型、不同 prompt 长度、不同并发数下的性能。
- 按任务复杂度做模型路由：
  - 简单同步任务走小模型。
  - 复杂异步任务走大模型。
  - 高价值任务可走远程强模型。

第二阶段再研究：

- continuous batching。
- KV Cache / PagedAttention。
- prefix cache。
- 量化推理。
- scheduler 参数。
- 本地推理与远程 API 的降级策略。

### 预期效果

- 项目从“调用远程模型”扩展到“可选择本地模型服务”。
- 能用数据说明不同模型和推理参数的延迟、吞吐和成本差异。
- 为后续 AI Infra / 推理方向学习建立项目入口。

### 面试表达

项目目前以 AI 应用和 Agent 工程为主。后续为了降低 API 成本、提升稳定性，并补充推理 Infra 能力，我计划接入 vLLM 本地推理服务。第一阶段会把部分低复杂度同步任务路由到本地模型，并记录 TTFT、TPOT、tokens/s 和 p95 延迟；第二阶段再研究 KV Cache、continuous batching 和量化推理。这样可以把业务 AI 应用自然延伸到模型服务和推理优化方向。
