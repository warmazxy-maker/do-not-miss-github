# Do Not Miss 待处理问题

## 001. 用户画像需要从规则关键词画像升级为 LLM 语义画像

状态：已处理，详见后端 `PROBLEM_LOG.md` 的 012 条。

### 当前情况

当前用户画像的原始来源包括：

- 已完成活动 / 已完成挑战，即个人成就历史记录
- 当前进行中的挑战数量
- 教练日志

后端目前会在推荐时读取这些数据，并通过 Java 规则进行聚合，生成一份结构化画像。画像字段包括：

- `summary`
- `strengths`
- `preferredCategories`
- `preferredLocations`
- `benefitPreferences`
- `evidenceKeywords`
- `recentSignals`
- `completedCount`
- `activeChallengeCount`
- `coachLogCount`

但这些字段主要来自统计、关键词匹配和模板拼接，不是 LLM 对用户经历进行语义总结后生成的画像。

### 发现的问题

当前画像更接近“规则画像 / 关键词画像”，表达能力有限：

- 只能识别预设关键词，难以理解用户经历背后的学习风格、长期目标、成长阶段和能力迁移。
- `summary` 虽然是总结形式，但本质上是 Java 模板拼接，不是语义归纳。
- 每次推荐时动态读取全部完成记录和教练日志，用户数据变多后会有性能压力。
- 个人成就模块虽然已经沉淀了用户成长数据，但还没有充分体现 LLM 对成长经历的分析价值。

### 优化方案

保留当前“原始来源”的更新逻辑不变：

- 用户完成活动后，写入成就记录。
- 用户完成挑战后，写入成就记录。
- 用户和教练对话并生成日志后，写入教练日志。

但最终用户画像不再主要依赖 Java 模板拼接，而是由 LLM 基于原始来源生成。LLM 不改变画像结构，只负责填充已有结构中的内容。

也就是说，画像结构仍然保持为：

```json
{
  "summary": "...",
  "strengths": ["..."],
  "preferredCategories": ["..."],
  "preferredLocations": ["..."],
  "benefitPreferences": ["..."],
  "evidenceKeywords": ["..."],
  "recentSignals": [
    {
      "type": "...",
      "title": "...",
      "category": "...",
      "detail": "...",
      "occurredAt": "..."
    }
  ],
  "completedCount": 0,
  "activeChallengeCount": 0,
  "coachLogCount": 0
}
```

LLM 的职责是根据真实完成记录、挑战记录和教练日志，生成更自然、更准确的画像内容，例如：

- 用户近期关注的方向
- 用户更适合的学习方式
- 用户已有能力基础
- 用户偏好的活动类型
- 用户可能需要补足的短板
- 用户成长轨迹中的稳定信号

### 推荐实现方式

不建议每次推荐时都调用 LLM 重新生成画像，也不建议每次教练聊天的每一句话都更新画像。

推荐采用“事件触发 + 异步更新 + 低频合并”的方式：

1. 用户完成活动或挑战时，写入成就记录，并标记用户画像 `dirty`。
2. 用户生成教练日志时，写入日志，并标记用户画像 `dirty`。
3. 后台异步任务定期扫描 `dirty` 用户，调用 LLM 生成新的画像快照。
4. 推荐接口优先读取画像快照，不再每次全量扫描历史。
5. 如果画像快照不存在或过期，再回退到当前 Java 规则画像。

可以新增一张画像快照表，例如：

```text
user_profile_snapshot
```

核心字段：

```text
user_id
summary
strengths_json
preferred_categories_json
preferred_locations_json
benefit_preferences_json
evidence_keywords_json
recent_signals_json
completed_count
active_challenge_count
coach_log_count
dirty
version
updated_at
```

### 预期效果

- 用户画像从“关键词提取”升级为“语义理解后的成长画像”。
- 个人成就模块真正接入 LLM，能够更像智能简历和成长档案。
- 推荐系统仍保持原有结构化画像输入，不需要大改推荐链路。
- 推荐接口性能更稳定，避免每次请求全量扫描历史数据。
- 面试表达上可以讲清楚：原始事实持久化、画像异步生成、推荐读取快照、LLM 输出结构化画像。

### 面试表达

当前项目的用户画像原始数据来自学生完成的活动、挑战和教练日志。后续优化中，我会将画像从规则关键词聚合升级为 LLM 语义画像：原始事实仍然由业务模块持久化，LLM 只负责在异步任务中根据真实记录生成结构化画像快照。推荐服务读取画像快照参与排序，如果快照不存在或生成失败，则回退到规则画像。这样既保证画像可解释和可追溯，又能让 LLM 发挥对成长经历的总结能力。

## 002. 成长曲线需要从固定标签折线图升级为智能标签与里程碑能力地图

状态：部分处理。已完成后端 `growth_tags` / `growth_tag_evidences` 数据模型、规则 fallback 标签生成、历史回填和基础查询接口；LLM 标签抽取、前端能力地图、标签详情里程碑 UI、智能简历引用仍待处理。

### 当前情况

当前个人成就模块中的成长曲线仍然是早期雏形：

- 后端预设固定成长维度，例如语言、研究、文化、沟通表达、内容创作等。
- 用户完成活动或挑战后，后端用规则把活动文本匹配到这些固定标签。
- 匹配成功后，对应维度加分。
- 前端用多条折线展示不同维度的增长情况。

这种方式可以证明“成长数据会随活动完成而变化”，但还不够像真实产品。

### 发现的问题

当前固定标签折线图存在几个问题：

- 固定维度过粗，不能表达每个用户独特的成长路径。
- 标签来自系统预设，不一定符合用户真实经历。
- 折线图颜色多、信息密度高，普通用户很难直观看懂。
- 用户只能看到分数变化，不容易回溯“为什么这个能力增长了”。
- 重要节点和普通节点没有明显区别，例如 CET-6、比赛获奖、项目上线等关键经历没有被突出。
- 智能简历生成时缺少“哪些经历更重要”的权重信号。

### 优化方案

将成长曲线从“固定标签 + 折线图”升级为“智能标签 + 能力地图 + 里程碑时间线”。

用户完成活动或挑战后，个人成就模块继续接收真实完成记录，但不再只用 Java 规则匹配固定标签，而是让 LLM 根据活动内容、挑战目标、用户填写的 `did` 和 `learned` 自动生成能力标签。

标签生成逻辑：

1. 用户完成一个活动或挑战。
2. 后端保存成就记录。
3. LLM 根据活动标题、内容、技能、用户复盘内容生成 1-3 个能力标签。
4. 如果标签已存在，则给已有标签加分并追加一条成长记录。
5. 如果标签不存在，则创建新标签。
6. 标签可以有别名归并，例如“Java 后端”“Spring Boot”“后端开发”可以被合并到相近能力簇。

用户最终看到的不再是多条折线，而是一个围绕个人头像展开的能力标签地图：

- 做得越多、证据越强的标签越大。
- 最近增长明显的标签可以更亮。
- 重要经历影响更大的标签可以更突出。
- 用户可以点进某个标签查看详细成长轨迹。

### 标签详情页设计

点击某个能力标签后，不展示普通列表，而展示类似“公路里程碑”的时间线：

- 每个里程碑代表一次相关活动、挑战或日志。
- 里程碑按时间排列。
- 悬浮到某个节点时展示：
  - 日期
  - 活动 / 挑战标题
  - 简介
  - 用户做了什么
  - 用户学到了什么
  - 标签贡献分
- 重要节点使用更醒目的样式。

例如用户通过 CET-6，可以在“英语能力”标签下形成一个突出里程碑。

### 重要事件标记

需要支持“重要事件”机制。

重要事件来源可以有两种：

- 用户手动标记，例如“这次比赛获奖很重要”“这次考过 CET-6 很重要”。
- LLM 自动识别，例如识别证书、获奖、项目上线、长期挑战完成、关键面试经历等。

重要事件会产生几个影响：

- 在能力地图中提高相关标签权重。
- 在标签里程碑中使用更醒目的展示。
- 在智能简历生成时优先作为核心经历参考。
- 在自我分析中作为阶段性成长节点。

### 推荐数据结构方向

可以新增能力标签表：

```text
growth_tags
```

核心字段：

```text
id
user_id
name
normalized_name
description
score
evidence_count
importance_score
last_updated_at
created_at
```

可以新增标签证据表：

```text
growth_tag_evidences
```

核心字段：

```text
id
user_id
tag_id
source_type
source_id
title
summary
did
learned
score_delta
is_milestone
milestone_reason
occurred_at
created_at
```

### LLM 的职责

LLM 不直接决定用户“厉不厉害”，而是负责从真实经历中抽取标签和解释证据。

LLM 输出建议保持结构化：

```json
{
  "tags": [
    {
      "name": "Java 后端开发",
      "normalizedName": "后端开发",
      "scoreDelta": 8,
      "evidenceSummary": "完成 Java 初级项目，实践了接口开发和项目结构设计",
      "isMilestone": false,
      "milestoneReason": ""
    }
  ]
}
```

后端仍然需要做校验：

- 标签数量上限
- 标签名称长度限制
- 标签归并
- 分数上下限
- 重要事件识别结果不能完全相信 LLM，需要允许用户修改

### 预期效果

- 个人成就模块从“统计图”升级为更贴近用户认知的“能力地图”。
- 用户既能看到整体画像，也能点进具体标签看成长证据。
- LLM 能参与个人成就模块，不只是推荐事件。
- 智能简历生成可以优先引用重要里程碑，表达更有说服力。
- 面试中可以讲成“从固定规则成长曲线升级为 LLM 驱动的能力标签系统”。

### 面试表达

当前成长曲线是固定维度的规则统计，能证明数据闭环，但表达能力不足。后续我计划将它升级成 LLM 驱动的智能能力标签系统：用户完成活动或挑战后，LLM 根据真实经历抽取能力标签和证据，后端进行标签归并、分数校验和里程碑标记。前端不再展示花哨折线图，而是展示围绕用户头像的能力标签地图，点击标签后可以查看按时间展开的成长里程碑。这样既提升用户体验，也让智能简历生成有更可靠的证据来源。

### 当前处理进度

- 已完成后端第二阶段：完成活动、挑战、复盘后自动生成或更新成长标签证据，并支持历史成就记录 rebuild。

- 已完成前端第四阶段：个人成就页新增能力地图，展示智能标签、累计分数、证据数量和重要性分，并支持手动刷新旧数据标签。
- 已完成前端第五阶段：点击能力标签后懒加载标签详情，展示成长证据里程碑时间线，并支持手动标记/取消重要里程碑。
- 已完成后端第六阶段：智能简历生成接入高分能力标签、重要里程碑和最近成长证据，LLM 与本地 fallback 都会优先引用这些结构化证据。
- 当前主线已完成：后续可以继续做 UI 细节、面试文档补充，或把 Coach 日志筛选后沉淀为成长标签证据。
